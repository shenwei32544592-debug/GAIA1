## 一、为何需要这份说明

艺术家联盟的联盟宪章与成立理由书，回答了**为何联合**、**权利从何而来**。  
本说明回答另一个问题：**这一切在 IT 和物理层面，为何可信、为何可落地？**

**本说明写给谁**：对 IT 技术了解有限、甚至完全不懂的读者——包括创始人自己在内。  
**本说明做什么**：替你做技术层面的思考与解释；用国家档案、银行流水、法院存证等现实世界已有先例作类比，说明艺术家联盟采用的技术类别是**已被验证可行的**，数据安全可控、协作可持续、技术服从宪政治理。你**有知情权**知道这些；但你**不必因不懂技术细节而担心或怀疑**——就像普通公民不必懂档案馆的恒温恒湿技术，也能信任档案不会丢失。

**常见疑虑**：  
「源头不可抹除」——平台改数据怎么办？  
「分配引擎无人可改」——谁说了算？  
「灵肉分离」——听上去抽象，实际是什么？  
「用不用区块链？技术靠谱吗？」  
「我不懂技术，怎么相信这一切？」

**本说明的目的**：用通俗语言和类比，逐一回答上述问题；说明承诺在技术上**如何实现、如何可验证、为何可信**，而非展开实现细节。

**读完本说明，你将理解**：  
(1) 源头、分账、灵肉分离在技术上如何实现；(2) IT 如何服从宪政治理（文官统军）；(3) **各节点如何参与制约 IT**——付款时的多方共管机制，不让任何人钻空子；(4) 采用的技术类别在国家档案、银行、法院中早有先例，非实验性；(5) Web3/区块链是可选工具，非必需；(6) 民众有知情权，但不必因不懂技术而担心——信任来自制度设计，不来自理解技术细节。

## 二、源头不可抹除：技术如何实现

**宪章原则**：创意主权、署名权、历史溯源优先于当前所有权。

**公众疑虑**：平台会不会偷偷改掉我的署名？历史会不会被覆盖？

**技术实现逻辑**（通俗表述）：

| 疑虑 | 技术手段 | 通俗类比 |
|------|----------|----------|
| 署名被篡改 | 系统设计为：署名记录**只允许追加，禁止覆盖**。任何修改都作为新记录追加，旧记录永久保留 | 像银行流水：每笔交易记一笔，不能回头删掉某一条 |
| 历史被抹除 | 数据存储采用**版本不可删除**策略；所有变更留痕 | 像国家档案馆：原件封存，修正只能以新文档形式追加 |
| 平台消失怎么办 | 规矩与档案 与 商业实例**分离**；数据可按标准格式导出、迁移 | 见第四节「灵肉分离」 |

**可验证性**：  
艺术家、裁定者、公益节点均可查询历史记录；若发现「覆盖」而非「追加」行为，即构成对联盟宪章的违反，可启动共识裁定。

## 三、分配引擎无人可改：技术如何保证

**宪章原则**：65/5/20/10 等参数由创世者锁定；分配引擎按规则执行，任何人不得 override。

**公众疑虑**：平台会不会偷偷改分成比例？创世者会不会单方加码？

**技术实现逻辑**（通俗表述）：

| 疑虑 | 技术手段 | 通俗类比 |
|------|----------|----------|
| 参数被悄悄修改 | 分润参数**写入系统配置**，变更须经创世者书面同意及多签；变更记录可查 | 像宪章修正案：有流程、有记录、可追溯 |
| 单笔交易被人为改写 | 分配引擎按 **RuleID** 自动执行；会计与结算系统**服从 RuleID**，无管理员 override 接口 | 像自动售货机：投币后按设定吐货，无人可中途改结果 |
| 谁监督执行 | 每笔分配有**账本记录**；公益金池、治理基准池等**全平台可查** | 像阳光下的财政：人民须看好钱袋子 |

**可验证性**：  
任何人可查阅分配规则版本、公益金池余额与流水；若发现未经流程的变更，即构成违规。

## 四、灵肉分离：架构上如何实现

**宪章原则**：GAIA 管规矩与档案（灵魂）；signa.ink、WhoAmI.Art 等管生意（肉体）。实例可替换，规矩与数据不随之消失。

**公众疑虑**：「灵魂」和「肉体」分开，具体是什么意思？

**技术实现逻辑**（通俗表述）：

| 概念 | 技术含义 | 通俗类比 |
|------|----------|----------|
| 灵魂层（GAIA） | 联盟宪章、创世档案、RuleID、署名与溯源数据结构；不依赖任一商业平台存活 | 像一国法律：政府可换届，法律文本与档案独立存在 |
| 肉体层（实例） | 对接支付、物流、获客的商业系统；负责赚钱与履约 | 像政府执行部门：按法律办事，可重组、可替换 |
| 数据可迁移 | 数据结构按**跨系统可迁移**设计；实例倒闭时，数据可导入新实例 | 像银行账户可跨行转账：你的存款不绑定某家银行 |

**可验证性**：  
若 signa.ink 因监管或攻击消失，GAIA 的联盟宪章、创世档案、艺术家与用户数据仍可经标准流程迁移至新实例，实现永续经营（Going Concern）。

## 五、物理载体：工业基底与自由源头

**创意载体规范原则**：物理载体必须臣服于精神印记；工业基底（Order）≥ 0.15mm；自由源头（Flow）严禁算法平滑。

**公众疑虑**：黄铜笔上刻的签名，凭什么说是「真」的？算法会不会偷偷改了我的笔触？

**技术实现逻辑**（通俗表述）：

| 概念 | 技术/物理含义 | 通俗类比 |
|------|---------------|----------|
| 工业基底 0.15mm | 线宽、路径闭合等**物理可制造底线**；低于此无法在黄铜等载体上成型，系统必须拦截 | 像建筑规范：承重墙有最低厚度，低于则不能通过验收 |
| 自由源头 | 艺术家的**原始笔触**（飞白、抖动、力度）；系统**严禁**算法「平滑」「美化」；符合工业基底则须 **100% 还原** | 像指纹：机器只负责识别与复刻，不能「优化」成别人的指纹 |
| 系统行为边界 | 系统可**检测、预警、驳回**；严禁**生成创意路径、替代人类创作、未经人类操作修改灵韵层** | 像质检员：可说不合格，不能替你重画 |

**可验证性**：  
艺术家可对比源头文件与物理成品；若发现算法擅自修改笔触形态，即违反创意载体规范，可启动审查。

## 六、共识裁定：技术如何支持

**宪章原则**：争议由同行共识裁定；无永久法官；公众可指认源头（PAC）。

**公众疑虑**：裁定会不会被平台或创世者单方操纵？

**技术实现逻辑**（通俗表述）：

| 环节 | 技术支撑 | 通俗类比 |
|------|----------|----------|
| 谁有权裁定 | 裁定者从**高贡献节点**中选取；按议题临时组陪审团，**议题结束即散** | 像陪审团：随机抽取、一案一组、无永久法官 |
| 公众如何参与 | 任何人可发起 **PAC**（公众源头指认）；提案进入共识流程 | 像公民提案：可发起，须经流程审议 |
| 裁决是否可被覆盖 | 共识结果写入**不可覆盖的账本**；无「管理员一键裁决」接口 | 像法院判决：落槌即入档，不能事后悄悄改 |

**可验证性**：  
裁定流程、证据链、结论均可查询；若存在未经共识程序的「裁决」，即违规。

## 七、宪政治理统辖 IT：技术层如何臣服于治理层

**核心问题**：IT 部门用哪些技术，保证互联网协作安全持续、数据安全可控，且**技术执行必须听从宪政治理**——就像宋代武官必须听从文官的制度设计？

### 7.1 文官统军的类比

宋代「文官统军」：武官掌兵，但调兵、决策权在文官；武官不能凭武力推翻文官决策。

**艺术家联盟的对应**：

| 角色 | 类比 | 约束关系 |
|------|------|----------|
| 宪政治理层 | 文官 | 制定规则；决定「能做什么、不能做什么」 |
| IT 执行层 | 武官 | 实现规则；**无权定义规则、无权覆盖规则** |

**技术上的保证**：不是靠「IT 人员自觉」，而是靠**系统设计**——把宪政约束写进架构，让违反约束的操作在技术上**无法执行**。

**IT 系统的多角色与高透明**：艺术家联盟的 IT 系统采用**多角色、高透明**设计——在数据层和代码层，均有专业人员参与监督；**无人能单独协调改动**，任何关键变更须经多签或约定程序。此设计与中央银行结算系统类似：多方共管、职责分离、操作留痕，从而具备高安全性的可实现基础。参见央行结算系统的多中心、多节点校验与审计机制。

**联盟为何能与时俱进**：联盟之所以能持续迭代、让人感到高效先进，是因为有强大的 CTO（技术节点）负责落实**任何节点提出的 IT 建设需求**——谁有想法、谁提需求，技术有人对接、有人落实；不因技术缺位而停滞。

**法律层面的追索机制**：管理法规中明确约定——**非授权改动**将被视为违规：违规者在联盟内的利益可被清算，且在现实世界中可被起诉追责。技术约束 + 法律追索，共同构成「没人敢、没人能」的双重威慑；这正是 IT 基底成为**最可信原因之一**的依据。

### 7.2 IT 将使用的技术类别（保证安全、持续、可控）

以下为**技术类别**而非具体产品名；工程实现时可选用符合这些类别的成熟方案。

| 目标 | 技术类别 | 通俗说明 |
|------|----------|----------|
| **数据不可篡改** | 只追加存储（Append-Only） | 署名、账本等核心数据只允许新增记录，禁止覆盖、删除；违反则系统拒绝执行 |
| **传输与存储安全** | 加密（Encryption） | 数据传输用 HTTPS 等加密通道；敏感数据存储加密；防止窃听与泄露 |
| **谁有权做什么** | 权限与能力隔离（Access Control） | 每个节点按身份获得「能力面具」；CTO 无分配参数修改权；管理员无「一键裁决」接口 |
| **操作可追溯** | 审计日志（Audit Trail） | 所有关键操作留痕；谁、何时、做了什么，可查；覆盖日志即违规 |
| **关键变更须多方同意** | 多签（Multi-Signature） | 分配参数、联盟宪章修正等须经创世者书面同意及约定多签；单方无法完成；见下文「多签机制通俗解释」 |
| **业务层服从规则层** | RuleID 驱动 | 分配引擎按 RuleID 自动执行；会计、结算系统**读取 RuleID、执行分配**，无 override 入口 |
| **实例可替换、数据不丢** | 数据可迁移设计 | 规矩与档案 与 实例分离；数据格式标准化；实例倒闭时，数据可导出、迁移至新实例 |
| **持续可用** | 备份与冗余 | 核心数据多副本；关键服务可故障切换；保障协作不因单点故障中断 |

### 7.2.1 多签机制通俗解释

**多签（Multi-Signature）是什么？**

通俗说：**一件事必须经多方同时同意，才能生效；缺一不可。**

**现实世界类比**：

- **公司银行账户**：大额转账须总经理 + 财务总监两人签字，少一个，银行不办。
- **保险柜**：双钥匙或双密码，两人各持一把/各知一段，缺一不可才能打开。
- **核按钮**：多人各自持有密码的一部分，必须一起操作才能发射——单方无法完成。

**艺术家联盟里的多签**：

分配参数（65/5/20/10 等）若要变更，须经**创世者书面同意**，并视条款级别同步履行：

- **多签参与方**：创世者 + CTO + 法务/财务（或约定的其他多方）
- **逻辑**：创世者说「我同意改」，还不够；CTO、法务/财务等也须签字确认，变更才能写入系统
- **效果**：创世者不能背着别人单方改参数；CTO 不能背着创世者改；任何一方想钻空子，其他方不签，就改不了

**实现方式**（可选，工程团队选型）：

- **传统方式**：纸质或电子变更单，各方签字后存档；系统管理员凭「已获多签的变更单」执行更新，审计可查
- **技术方式**：多签钱包、多签审批流程、密钥分持等——具体由技术团队根据成熟度选定

**核心**：不是「谁权力大谁说了算」，而是**制度规定「必须多少人同时同意」**；少一个，变更无效。

| 联盟宪章约束 | 技术实现方式 |
|----------|--------------|
| 源头不可抹除 | 署名、溯源表**禁止 UPDATE**；只允许 APPEND；数据库层或应用层强制 |
| 分配引擎无人可改 | 分配参数存入**只读配置**；变更走多签流程；分配逻辑与参数分离，CTO 只实现逻辑，不持有参数写入权 |
| 无管理员一键裁决 | 系统**不提供**「管理员直接改结果」的 API；裁决结果只能经共识流程写入 |
| 历史不可覆盖 | 版本化存储；删除、覆盖操作在代码层面**被拒绝** |
| CTO 无权改参数 | 权限设计中，CTO 角色**无**分配参数写入能力；创世者持有；变更须书面流程 |

### 7.3 制度设计的 IT 可行性：递归反哺、待认领池、公众指认

**常见疑虑**：想法很好，无法实现也只能是空想。递归反哺、行业始祖 0.1%、待认领池、公众指认——这些制度在技术上能不能落地？

**结论**：**可落地**。均为标准账务 + 治理决议驱动，无实验性技术依赖。

| 制度环节 | IT 实现逻辑 | 通俗类比 |
|----------|-------------|----------|
| **0.1% 分流** | 订单分配时，按衍生创作者声明的品类，将 0.1% 计入该品类待认领子账户 | 像电商：每笔订单按规则拆成卖家款、平台费、退款池——纯账务逻辑 |
| **待认领池** | 按品类分账的表结构（如 `ancestor_escrow(category_id, amount, ...)`）；余额可查询、可审计 | 像代管金：钱先放着，等有决议再划出 |
| **指认→支付始祖** | 经公众指认确认后，治理决议触发：系统将待认领余额 + 未来 0.1% 路由至始祖账户 | 像退款：买家确认收货后，系统按规则把钱从 escrow 打给卖家 |
| **公投→转入公益** | 行业内公投通过后，治理决议触发：系统将待认领余额从 escrow 转入公益金池 | 像慈善捐赠：董事会决议后，财务按决议划款 |
| **公众指认（PAC）** | 提案提交 → 进入共识流程 → 裁定者裁定 → 结论写入不可覆盖账本 | 像公民提案：可发起，须经流程审议；技术支撑流程与留痕 |

**无根本实现障碍**；与银行流水、电商分账、公益基金托管同类。表结构与实现方向见《递归反哺与源头确权细则》§5.8.6。

### 7.4 小结：技术臣服于治理

**宪政治理定义「什么合法、什么非法」；IT 实现「让非法的在技术上无法执行」。**

- 文官（治理层）说：武官不得擅自调兵。  
  技术保证：调兵接口需多签，单方调用无效。

- 宪政说：署名不可覆盖。  
  技术保证：数据库约束 + 应用逻辑，UPDATE 署名字段即报错。

- 宪政说：分配引擎无人可 override。  
  技术保证：分配引擎无管理员入参；会计系统只读 RuleID，不接收人工改写指令。

**这不是信任 IT 人员，而是用制度把宪政约束编译进系统**——技术层在架构上即已臣服于治理层。

### 7.5 Web3、区块链是什么，GAIA 用不用？

**区块链 / Web3 是什么（通俗解释）？**

可以把它想成一本**大家共有的账本**：每发生一件事，就往上记一条，记上去就不能再改、不能删。很多人各留一份副本，谁想偷偷改自己那份，会跟别人的对不上，立刻被发现。所以「只追加、不可篡改」是它天生的特性。

**GAIA 需要的能力， blockchain 恰好有；但 GAIA 未必非用 blockchain 不可。**

| 宪政需要的能力 | 区块链天生具备 | 传统技术也能做到 |
|----------------|----------------|------------------|
| 只追加不覆盖 | 链上每笔都是新区块，不能回头改 | 数据库设约束：禁止 UPDATE/DELETE；违反即报错 |
| 多方可验证 | 链上数据公开或按权限可查 | 审计日志、开放查询接口 |
| 不可单方篡改 | 需多数节点同意才能写 | 多签；权限隔离；CTO 无写权限 |

**所以**：区块链是一种**可选工具**。如果工程团队评估后认为它成熟、合规、成本可承受，可以采用；如果认为传统数据库 + 严格约束更稳妥，也可以。**宪政原则不变——要的是「只追加、不可篡改、可审计」，不是「必须用某一种技术」。**

**术可与时俱进**：今天用 A 技术，五年后若有更好的 B 技术，可以在不违背宪政的前提下迁移。国家档案从纸质到电子、从单机到云端，也是这样演进的。

### 7.6 术的层面：现实世界早有先例，民众有知情权但不必因不懂而担心

**这些技术在现实世界中是否已被验证？**

是。**国家、银行、法院、企业，早已用同类技术支撑协作与存证。** 艺术家联盟采用的，不是实验性概念，而是已被证明可行的技术类别。

**国家档案库的类比**：

国家有档案馆，存法律文书、司法卷宗、历史记录。公民不必懂防磁防潮、温湿度控制、数字化格式，也能信任：档案不会轻易丢失，不会被某个人悄悄涂改。**为什么？** 因为：(1) 有法律规范怎么存、怎么取；(2) 有专职人员按规程操作；(3) 有审计和抽查。公民的信任，建立在**制度 + 专业 + 可追查**上，而不是自己懂技术。

**艺术家联盟的对应**：

| 国家档案 | 艺术家联盟 |
|----------|------------|
| 法律规范存管 | 联盟宪章、创意载体规范规定什么可做、什么禁止 |
| 专职人员按规程 | IT 按宪政约束实现；权限隔离，越权操作被拒绝 |
| 审计可查 | 审计日志、全平台可查的公益金池、分配记录 |

**银行流水的类比**：

你查银行 App 的余额和流水，不会担心银行偷偷改数字。你并不懂数据库怎么存、备份有几份、怎么防止内部人篡改。你信任的，是**监管 + 审计 + 法律追责**这套制度。艺术家联盟的分配引擎、RuleID、全平台可查账本，逻辑相同：**制度保证可查、可追、越权即违规**。

**知情权：你可以知道什么？**

| 你有权知道 | 具体可查 |
|------------|----------|
| 有规则约束技术 | 联盟宪章、创意载体规范、CTO 共识协议明文禁止哪些行为 |
| 数据有保护 | 加密传输、权限隔离、只追加不覆盖——技术团队按这些类别实现 |
| 关键操作可追溯 | 分配规则变更、公益金出入、共识裁定结论，均有记录 |
| 数据不会随平台消失 | 灵肉分离；数据可迁移；实例可换，档案不丢 |

**不必因不懂而担心**：

你不需要懂 HTTPS 的加密原理，也能相信网页密码不会被中途截获；不需要懂数据库锁，也能相信银行不会偷偷改流水。**信任来自制度设计，不是来自你理解技术细节。** 艺术家联盟亦然：宪政与创意载体规范定义了「道」；技术是「术」。你只需确认：道是对的、术是现实世界已验证的类别、有制度保证术服从道。具体用哪种数据库、是否上链，是工程选型问题，**普通民众有知情权知道「用了哪几类技术、为何足够」，但不必因不懂具体实现而怀疑**。

**道与术的区分**：

- **道**（不变）：宪章原则、创意载体规范、源头不可抹除、分配引擎无人可改。这些不因技术换代而改变。
- **术**（可演进）：具体用 PostgreSQL 还是区块链、用哪种加密算法、用哪家云服务。国家也会升级档案系统、引进新存证方式；艺术家联盟的技术选型同样可以与时俱进，只要不违背道。

### 7.7 可能出什么问题？谁在监督技术？

**如果不懂技术，自然会问：万一出事怎么办？谁保证 IT 不会乱来？**

**可能的风险与应对**：

| 风险 | 应对 |
|------|------|
| 平台被攻击、数据泄露 | 加密存储；权限隔离（攻击者拿不到全部）；审计日志可追溯泄露源 |
| 平台倒闭、服务中断 | 灵肉分离：规矩与档案 与 数据可迁移至新实例；你的署名、分润权不绑定某一家公司 |
| IT 人员擅自改数据 | 权限设计：CTO 无分配参数写入权；署名只追加不覆盖，改即违规；审计日志留痕 |
| 某笔分账被人工改写 | 分配引擎无 override 接口；会计系统只读 RuleID；若发现异常可启动共识裁定 |

**谁监督技术？**

- **宪政治理层**（创世者、门类理事会、裁定者）定义规则；技术层实现规则，**无权修改规则**。
- **共识裁定**：若有人指控技术违规（如篡改署名、擅自改分配），可启动裁定者陪审团调查；审计日志、代码审查可提供证据。
- **全平台可查**：公益金池、分配记录、规则版本——任何人可查；异常会被发现。
- **制度优于信任**：不依赖「相信某个技术人员是好人」，而依赖「他即使想乱来，系统也不提供那个入口」。

**总结**：技术风险有对应预案；技术层受治理层约束、受审计约束、受共识裁定约束。普通民众不必懂具体实现，只需知道**有制度在兜底**。

### 7.8 各节点如何参与制约 IT：付款时的多方共管机制

**核心问题**：不是只有 IT 和创世者在管钱；**艺术家、门类理事会、裁定者、执行节点、联盟代表会**——各方如何参与制约？付款时如何**多方共管**，让任何人钻不了空子？

#### 付款流程中的角色分工（谁管什么、谁不能管什么）

| 角色 | 在付款/分配中的权力 | 制约方式 |
|------|---------------------|----------|
| **创世者** | 锁定 65/5/20/10 等参数；参数变更须书面同意 | 变更须**多签**（Founder + CTO + 法务/财务）；创世者**不得修改已生效分配规则** |
| **CTO / 技术节点** | 实现分配引擎代码 | **无参数写入权**；只实现逻辑，不持有修改比例的权限 |
| **执行节点（LLC）** | 代收代付；对接 Stripe/Mercury；按指令打款 | **会计系统必须服从 RuleID**；无权改分配比例；禁止利润滞留；银行流水对联盟代表会**公开**；联盟代表会可**撤回授权**、24h 内换执行节点 |
| **分配引擎** | 按 RuleID 生成分配指令 | **无人可 override**；无管理员改结果接口 |
| **门类理事会** | 管准入，不碰钱、不碰码 | 不参与付款，但可**查阅**；若发现分配异常可提议调查 |
| **裁定者** | 裁定争议 | 若有人指控分配被篡改、执行节点截留，可启动**共识裁定**；审计日志、流水可作证据 |
| **艺术家 / 用户** | 收款方；可查自己分润 | **分润明细可查**；公益金池**全平台可查**；发现异常可发起 PAC 或提请调查 |
| **联盟代表会** | 签署《自然人宪章》的联合体；授权执行节点 | 可**撤回**执行节点授权；执行节点是**代理人（Agent）**，不拥有分配决策权 |

#### 付款时的多方共管：钱怎么流、谁动不了

```
用户付款 → Stripe/Mercury 收钱 → 执行节点（LLC）代收
                    ↓
           分配引擎按 RuleID 生成分配指令（65%艺术家 / 5%创世分润池(2%+2%+1%) / 20%运营池 / 10%储备金）
                    ↓
           执行节点会计系统【只读 RuleID，按指令执行】——无 override 入口
                    ↓
           打款至艺术家、各池、公益金账户
```

**关键约束**：
- 执行节点**不能**自己决定「这笔钱我少给艺术家 10%」——系统不提供这个入口；它只能按 RuleID 执行。
- CTO**不能**改分配比例——权限设计上无此能力。
- 创世者**不能**单方改参数——须多签。
- 任何人想「钻空子」改单笔分配——**没有接口**；想改就得改 RuleID 或参数，而那两条路都有制约。

#### 堵漏洞：若有人想钻空子，怎么防？

| 想钻的空子 | 防护机制 |
|------------|----------|
| 执行节点截留、少给艺术家 | 分配比例硬编码；会计服从 RuleID；银行流水对联盟代表会公开；禁止利润滞留（授权书明文） |
| CTO 偷偷改比例 | 权限隔离：CTO 无参数写入权；变更须多签 |
| 创世者单方加码 | 多签流程；变更记录可查；联盟宪章约束「不得修改已生效分配规则」 |
| 某人 override 单笔分账 | 系统**不提供** override 接口；分配引擎无人工入参 |
| 执行节点把 RuleID/域名列为公司资产 | 授权书禁止；联盟代表会可撤回、换节点 |
| 某池（如公益金）被挪用 | 公益金池全平台可查；动用权分层、有协议约定；异常可被任何人发现并提请调查 |
| 紧急状态被滥用 | 14 天披露期；75% 共识；7 天生效；每次≤6 月、续延≤3 月；12 月冷却期；60 天内裁定者事后审查；无决议不划款 |
| 固定雇佣吃掉 FCR 结余 | 无人受雇原则：全外包、按单计费；法定签字人为受托人非雇员；成本随业务量波动 |

**总结**：付款不是「一个人或一个部门说了算」。创世者锁参数、多签才能改；CTO 只实现、不能改参数；执行节点只执行、不能改指令；艺术家和公众可查、可质疑；裁定者可裁定。**多方共管，相互制约，不让任何人钻空子**。

### 7.8.1 现实法律层面：谁承担与收款壳公司、资金池的相应关系？

**核心问题**：银行账户、Stripe、资金池——在法律上，哪个实体或个体是账户持有人？谁对银行、税务、合同负责？

#### 法律结构概览

| 层级 | 法律定性 | 现实中的对应 |
|------|----------|--------------|
| **联盟代表会** | 由《自然人宪章》签署人组成的联合体；**非法人联合体**（不构成公司、合伙企业） | 创始人、CTO、门类理事会成员等**自然人**签署宪章后形成；他们是制度的**委托人（Principal）** |
| **执行节点（LLC）** | 联盟代表会的**代收代付代理人（Agent）**；在特定法域（如美国德拉华州）设立的公司 | 实际持有银行账户（Mercury）、Stripe 账户；与银行、支付机构**签约的主体**；资金在此实体名下流转 |
| **资金池** | 逻辑上的分账结构（艺术家 65%、治理 5%、运营 20%、储备 10%、公益金） | 资金**实际存放**于执行节点 LLC 的银行账户（或该 LLC 名义下的子账户、信托账户）；账本记录各池权重与流水 |

#### 谁对谁负责？

| 关系 | 法律上的承担方 |
|------|----------------|
| **与银行、Stripe 的合同** | 执行节点 LLC——它是账户持有人、签约方 |
| **与税务当局** | 执行节点 LLC 申报、纳税；若为单成员 LLC，成员（如创始人）可能涉及穿透申报，视法域而定 |
| **对艺术家、用户的付款义务** | 执行节点 LLC 按 RuleID 执行打款；若 LLC 违约（如截留），联盟代表会可撤回授权、换节点，并依授权书追究 |
| **对联盟代表会的服从义务** | 执行节点 LLC 作为 Agent，须服从授权书；银行流水对联盟代表会公开；联盟代表会可撤回授权 |
| **公益金池等托管** | 递归反哺细则：托管主体为「平台（LLC）或指定合规主体」；决策权与执行权分离——治理主体决定用途，平台按决议划款 |

#### 通俗类比

- **联盟代表会** = 业主；**执行节点 LLC** = 物业公司。业主委托物业代收物业费、代付维修款；物业的银行账户里有钱，但物业**不能擅自决定**怎么分——必须按业主约定的规则（RuleID）执行。业主可随时换物业。
- **资金池** = 物业账户里的「虚拟分账」：多少归 A 楼、多少归 B 楼、多少入储备金——实际钱在同一个或关联的银行账户，账本记录每笔归属。

#### 重要澄清

- 联盟代表会**不是**法人，不直接持有银行账户；自然人通过宪章联结，授权 LLC 代为执行。
- 执行节点 LLC 是**工具**——「公司只是被宪章调用的工具」；联盟代表会通过授权书约束其行为，通过撤回权保持控制。
- 具体法人形态（单成员 LLC、多成员 LLC、未来公益信托等）由法务与税务按落地法域设计；**原则**是：执行节点为 Agent、联盟代表会为 Principal、资金池逻辑清晰、决策与执行分离。

*本段为解释性说明；具体法律结构以正式法律文件为准。*

### 7.8.2 长期以后：谁在现实世界中对外签字并产生法律效力？

**核心问题**：联盟代表会由自然人组成，非法人。长期以后，当创世者年迈、卸任或故去，**谁的签字能代表联盟代表会、产生法律效力？** 就像总统：当选且在任期内签字有效，卸任后无效。将来由什么样的组织机构，委托出什么样的自然人或机构，行使现实世界中的签字权与法人地位权利？

#### 当前阶段（创世期）的约定

- **授权书签署**：由「宪章签署人代表联邦向 LLC 发出本授权书」
- **实际操作**：创世者及首批 创世公民 签署宪章后，由其中**具备代表性者**（如创世者）作为联盟代表会的**授权签字人**，对外签署执行节点授权书等法律文件
- **依据**：联盟代表会内部约定（可写入宪章或配套协议）：谁有权代表联盟代表会签字、签字须经何种确认（如多签、会议决议）

#### 长期须建立的制度（已起草草案）

**原则**：签字权与法人地位的行使，必须来自**联盟宪章或自然人宪章规定的程序**，而非某人的自然继承或私下指定。谁「在任」、谁有权签字，应当**可查、可验证、可承继**。

**已起草草案**：[《签字权与法人地位承继规则（草案）》](../02_协议与规范/签字权与法人地位承继规则（草案）.html)

草案要点摘要：

| 制度要素 | 草案规定 |
|----------|----------|
| **创世期** | 创世者为主签字人；执行节点授权等须主签字人 + 至少 1 名 创世公民 联署；可指定 1–2 名备份签字人 |
| **长期** | 主签字人经 创世公民 推举或门类理事会+创世者提名，创世公民 过半数确认；任期 3 年，可连任 |
| **签字生效** | 关键文件须多签（主签字人 + 备份签字人/创世公民/门类理事会代表） |
| **承继** | 卸任须 90 天提前通知；30 日内完成继任程序；死亡/失能时备份签字人代行 |
| **法人受托** | 可选；经 创世公民 + 门类理事会决议可委托公益信托等法人；创世期不启动 |
| **公示** | 当前签字人名单须在 GAIA 官网可查 |

#### 与第六编的衔接

联盟宪章第六编规定：修正案须经技术审查、伦理评审、裁定者超级多数共识。**签字权与法人地位的承继规则**，可作为宪章配套协议，经 创世公民 与门类理事会确认后生效；重大修正可同步履行第六编 consensus 流程。文明延续条款（第六十六条）——「不以公司存续为前提，只以文明记忆为依托」——要求：**谁有权签字、谁代表联盟，必须由协议与共识决定，而非依附于某一家公司或某一个人**。

| 账户应急指令（ACI） | 可选存储表（emergency_contacts、beneficiaries）；录入 2FA；激活时证明上传、审计日志；标准 CRUD + 工作流 | 《账户应急指令规则（草案）》§7 |
