# 建国期人事网：自发协作结构设计（草案）

**用途**：在去中心化联盟前提下，让建国期具备「人事组织部」般的有序协作能力；使加入的节点在制度框架内**自发长出**有层级、有明确任务和权利的协作结构（人事网）  
**版本**：草案 v0.1  
**依据**：《创世贡献记录与确认机制》《GAIA 节点角色与 IT 平台映射》《GAIA 社区早期权重计算逻辑》《裁定者身份、遴选与退出细则》  
**原则**：制度预留位，节点填位；任务公开，认领制；贡献→资格→权利；层级自发涌现，非 Founder 任命

---

## 一、核心矛盾与设计取向

| 矛盾 | 设计取向 |
|------|----------|
| 去中心化 vs 建国期需协调 | **制度定骨架，节点填血肉**；协议定义「有哪些位子、什么权利」，人不指定 |
| 自发 vs 有序 | **任务公开 + 认领制**；谁认领、谁完成、谁被确认，全透明可见 |
| 层级 vs 非任命 | **贡献→资格→权利**；达到阈值自动具备某角色资格；层级由贡献与附和自然浮现 |
| 人事能力 vs 无传统人事部 | **论坛 + 看板 + 贡献记账**；人事网不是部门，而是**可查询、可追溯、可演进的协作图谱** |

---

## 二、人事网的骨架：制度预留位

联盟宪章与既有协议已定义的角色与位子，构成人事网的**空位**；节点通过认领、贡献、确认来**填位**。

### 2.1 层级与位子（按就位时间排序）

| 层级 | 位子 | 定义 | 填位方式 | 权利 |
|------|------|------|----------|------|
| **L0 内核** | 创世者、创世技术官 | 制度设计所需；已锁定 | 创始既定 | 执行权、72h 暂停、公益金决策（Founder）；技术架构（CTO） |
| **L1 治理候选池** | 创世公民、门类理事会成员、裁定者候选人 | 高贡献、可参与治理 | 90 日内贡献 + 确认人附和 + 共识/投票 | 治理投票、准入审核、裁定陪审资格 |
| **L2 任务负责人** | 各建"国"任务的认领者/责任人 | 认领并完成具体任务 | 论坛认领 + 完成 + 确认 | 该任务范围内的协调权、产出定义权 |
| **L3 贡献者** | FCR 贡献者、有效参与人 | 有实质贡献但未达 L1 | 贡献自报 + 附和确认 | 参与度可见、FCR 评估输入、权重积累 |
| **L4 围观/潜在** | 发言者、认领门类未签约者 | 参与但未形成可确权贡献 | 发言、附和、认领门类 | 被看见、参与度展示、转化通道 |

### 2.2 位子之间的演进路径

```
L4 围观 → 发言/认领门类 → 参与度可见
         ↓
L3 贡献者 ← 贡献自报 + 附和确认
         ↓ (90日内实质贡献 + 权重达标)
L1 治理候选池 ← 创世公民签约 / 门类理事会就位 / 裁定者池入选
         ↓
L2 任务负责人 ← 认领建"国"任务 + 完成 + 确认
```

**关键**：L2 可与 L1/L3 重叠；一节点可同时为创世公民 + 某任务责任人。层级描述的是**资格与权利类型**，非行政隶属。

## 三、任务公开与认领制

### 3.1 建"国"任务清单（已有，见创世贡献机制 §3.2）

创世者在论坛开立「建"国"任务」专区，列出：

- 协议起草、技术模块、合规、财务、首单闭环、首批艺术家引入、门类开通、论坛搭建…
- 每一任务可设：责任人（**空缺待认领**）、截止意向、产出定义

### 3.2 认领规则

| 规则 | 说明 |
|------|------|
| **公开认领** | 节点在论坛发帖：「认领任务 X；预计产出 Y；截止意向 Z」 |
| **确认人附和** | 创世者或指定确认人回复「确认」或「+1」→ 认领生效 |
| **完成即留证** | 完成后在同一任务帖下回复：产出物、日期、可验证链接 |
| **一人多任务** | 允许；但建议单节点同时认领不超过 2–3 项，避免过载 |
| **让出与交接** | 责任人可发帖说明无法继续，开放重新认领；交接须有确认人见证 |

### 3.3 任务→权利映射

| 任务类型 | 完成后的权利/资格 |
|----------|------------------|
| 技术模块交付 | 计入 FCR；可进入裁定者池候选（若领域相关） |
| 艺术家引入（≥N 人签约） | 计入 FCR；拉人权重；可提名门类理事会 |
| 门类开通与运营 | 该门类理事会成员候选 |
| 合规、财务、法律 | 计入 FCR；专业能力标识，可参与相关裁定陪审遴选 |
| 论坛搭建、制度广场 | 计入 FCR；CTO 或 IT 贡献者；技术权重 |

**无节点能出力时的紧急采购**：建"国"任务中须对外采购但无内部资源的，可由[《建国者共建金》](建国者共建金：小额投入与紧急采购机制（草案）.md)支持。

## 四、贡献→资格→权利（量化与阈值）

### 4.1 与既有机制的衔接

| 机制 | 本设计中的用途 |
|------|----------------|
| **创世贡献记录（论坛）** | 贡献自报、附和、任务完成留证 |
| **Reputation Score** | 参与度、建议被采纳、有效邀请→权重；高权重→发言高亮、FCR 协商参考 |
| **FCR 评估** | 90 日结束后的比例约定；任务完成、附和、邀请人数为输入 |
| **裁定者遴选** | FCR 贡献突出 + 能力标识 + 共识/投票→进入裁定者池 |
| **门类理事会** | 门类认领 + 该门类任务完成 + 艺术家引入→理事会成员候选 |

### 4.2 资格阈值（示意，可经创世期共识微调）

| 资格 | 阈值（示意） | 说明 |
|------|--------------|------|
| 创世公民签约 | 认同宪章 + 创世者/确认人认可 | 已有流程 |
| 门类理事会成员 | 该门类认领者中贡献分 Top N + 共识确认 | 3–5 人/门类 |
| 裁定者池入选 | 贡献分达有效区间 + 能力标识 + 创世公民/门类理事会过半数确认 | 见《裁定者身份、遴选与退出细则》 |
| 任务责任人 | 认领 + 确认人附和 | 无贡献分硬门槛；完成度事后评估 |

## 五、人事网的可视化与透明

### 5.1 看板元素（建议）

| 元素 | 内容 | 目的 |
|------|------|------|
| **任务看板** | 任务清单、认领人、状态（待认领/进行中/已完成） | 谁在干什么，一目了然 |
| **节点图谱** | 按门类/任务/贡献分层展示；谁在 L1 池、谁在 L2 负责什么 | 人事网即协作图谱 |
| **参与度排行** | 发帖数、附和、采纳、邀请（脱敏或自愿公开） | 谁在积极参与，可见 |
| **门类认领数** | 各门类已认领人数、已签约艺术家数 | 火势可见；FOMO 促行动 |

### 5.2 实现层级（与创世贡献机制 Phase 1–3 对齐）

| Phase | 人事网可实现 |
|-------|--------------|
| **Phase 1** | 任务清单 + 认领帖 + 确认人附和；手工维护看板或简单表格 |
| **Phase 2** | 论坛标签/徽章：任务责任人、门类理事会、裁定者池；参与度、RP 展示 |
| **Phase 3** | 自动图谱：任务→责任人→贡献→资格；权重参数公开；裁定者池数量可查 |

## 六、自发涌现的保障机制

### 6.1 防止「占位不干活」

| 机制 | 说明 |
|------|------|
| **贡献分时效衰减** | 长期不参与→贡献分衰减→跌出裁定者池/门类理事会有效区间 |
| **任务让出** | 责任人久未更新可经确认人提醒；超期无产出可开放重新认领 |
| **退出通道** | 裁定者、门类理事会均有退出细则；无永久占位 |

### 6.2 防止「层级固化为阶层」

| 原则 | 说明 |
|------|------|
| **按案遴选** | 裁定者无永久法官；每次按领域+随机从池中抽 |
| **门类轮换** | 门类理事会可设任期（如 2 年）；届满可连任但须经共识 |
| **FCR 不绑定治理权** | 治理层进入资格基于角色与能力，非 FCR 持有（见 FCR 附录） |

### 6.3 中枢角色：不定人事，定规则与确认

| 中枢职责 | 不职责 |
|----------|--------|
| 发布任务清单、定义产出 | 不指定「谁必须是责任人」 |
| 对认领与完成进行确认、附和 | 不否决合规认领（除非明显冲突） |
| 组织盘点、共识、投票 | 不单方面任命门类理事会/裁定者 |
| 维护看板、促进透明 | 不隐瞒谁在什么位子 |

## 八、下一步行动

| 动作 | 责任方 |
|------|--------|
| 在论坛建「建"国"任务」专区，发首批任务清单（责任人空缺） | Founder |
| 在任务帖中明确认领格式、产出定义 | Founder |
| 首次认领发生时，确认人附和，并在看板中更新 | Founder / 指定人 |
| 90 日盘点时，将任务完成、贡献、附和纳入 FCR 评估输入 | Founder + 确认人 |
| 门类理事会、裁定者就位时，按贡献分+共识执行，不任命 | 创世公民 + 门类理事会 + Founder |
