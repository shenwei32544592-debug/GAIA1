艺术家与系统需求直连机制（草案）

全称：艺术家 × 系统需求区——制度与技术的输入端
用途：联盟的神经系统；确保创作者的真实需求第一时间进入技术与规则的演化过程；艺术家=感觉器官，IT=中枢神经，制度=肌肉与骨骼
版本：草案 v0.1
性质：协议附件，与《创世贡献记录与确认机制》《公民参与与贡献记账》配套
设计意图：不是功能，是联盟的生死开关；让艺术家被 IT 听见；需求不是线性的而是突变式的；公开压力场促成自进化

一、艺术家直连原则（可嵌入宪章/理由书）

艺术家直连原则

GAIA 承认： 一切有效制度，必须来自真实使用者的持续反馈。

因此，联盟设立「艺术家 × 系统需求」直连机制， 确保创作者的真实需求 能第一时间进入技术与规则的演化过程。

我们不承诺立即满足所有需求， 但承诺：不忽略任何一个源头的声音。

二、专区定义

2.1 名称与定位

项目 约定
专区名称 【艺术家 × 系统需求区】
副标题 给系统的真实声音
性质 艺术家 → IT 的专用直连通道；不是聊天区、吐槽区、情绪宣泄区
功能 制度与系统的输入端；需求暴露、冲突权衡、方案被反复否定的进化压力场

2.2 置顶须知（三点）

  1. 这里不是吐槽区
  2. 这里不是情绪宣泄区
  3. 这里是制度与系统的输入端——您的需求会进入技术与规则的演化过程

三、身份强区隔（核心设计）

身份 权限
艺术家 唯一可发主帖；需求由源头发出
IT、策展、运营 仅可回复、追问、解释；不可发主帖
围观者 仅可阅读；不可插话

设计意图：艺术家被尊重的感觉；需求质量大幅上升；无关讨论自动消失。

四、需求必被处理（灵魂条款）

每一个需求，须进入以下三种状态之一：

状态 说明 要求
已采纳 计划中 / 已实现 须有明确回应；可注明预计时间或任务编号
暂缓 原因说明 须说明暂缓理由（资源、优先级、依赖等）
拒绝 制度边界说明 须说明为何与制度/协议/技术约束不符

原则：艺术家真正需要的不是「你答应我」，而是**「我没有被忽略」**。哪怕一句话解释，都足够。

五、需求发帖模板（降低无效噪音）

发帖时建议包含（非强制，但有助于快速处理）：

字段 说明
需求简述 用一句话说清：您希望系统/规则做什么改变？
使用场景 在什么情况下遇到问题或产生此需求？
与制度/规则的关系 若涉及分润、递归、载体等，请注明相关条款
优先级自评 紧急 / 重要 / 可选（供 IT 排序参考）

六、IT 贡献权重激励

行为 贡献权重
IT 在本区的有效回应 计入 IT 的贡献权重（C 类共识贡献）
回应被艺术家点赞/引用 加分
需求被纳入制度或实现 加分；可计入「建议被采纳」

设计意图：反向驱动 IT 自发盯着此区；优秀系统是被真实使用逼出来的。

七、自进化逻辑

现象 机制
需求不标准、不预知、突变式 先画蓝图再让艺术家适应 → 系统必死；允许需求直接暴露、在冲突中权衡 → 能活下来的方案被反复使用
公开压力场 艺术家可见：哪些需求被回应、哪些被拒绝；IT 可见：哪些需求反复出现、哪些是个体噪音
共同语言形成 艺术家学会提「有效需求」；IT 学会解释「制度边界」；双方形成联盟语言
胚胎式生长 没有蓝图，却长出路径；感觉与中枢直连，痛点被立刻感知，错误被快速修正

八、与论坛结构的关系

本专区为论坛内独立子区，与建「国」路线图专区、任务清单专区并列。
身份区隔仅在本区生效；其他专区按《创世贡献记录与确认机制》约定执行。

九、生效条件

  1. 论坛具备身份识别能力（艺术家 vs 其他）
  2. 创世技术官或指定 IT 节点承诺响应时效（如 7 日内必有状态更新）
  3. 状态标签（已采纳/暂缓/拒绝）可被系统记录与公示
