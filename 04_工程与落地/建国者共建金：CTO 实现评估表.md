# 建国者共建金：CTO 实现评估表

**用途**：从 CTO 视角评估建国者共建金各模块的实现路径、工作量与困难度；供排期与选型决策  
**依据**：《建国者共建金：小额投入与紧急采购机制（草案）》《公益金池细则》《创世贡献记录与确认机制》

---

## 一、总览

| 阶段 | 实现路径 | 总工作量（人天） | 难度 | 核心依赖 |
|------|----------|------------------|------|----------|
| **Phase 1** | 手工录入 + 论坛公示 | 0–1 | 低 | 论坛就绪；账户开立（非 CTO） |
| **Phase 2** | 结构化录入 + 独立展示页 | 3–6 | 低–中 | 无（可独立于论坛） |
| **Phase 3** | 支付接入 + 治理工作流 | 8–15 | 中 | 支付资质、治理后台 |

**说明**：Phase 1 几乎零开发，Founder 手工维护即可；Phase 2/3 为体验与效率升级。

## 二、分模块评估

### 2.1 入账（参与人投入）

| 模块 | Phase 1 | Phase 2 | Phase 3 | 工作量(人天) | 难度 | 说明 |
|------|---------|---------|---------|--------------|------|------|
| **收款渠道** | 银行转账/支付宝/微信→人工确认到账 | 同左 | 在线支付（Stripe/支付宝/微信）→自动入账 | 0 / 0 / 5 | — / — / 中 | Phase 1/2：参与者转账后，Founder 核对到账并录入；Phase 3 需支付 SDK |
| **入账记录** | 论坛专帖或表格，手工追加「参与人、日期、金额」 | 表单提交→写入 JSON/DB→展示 | 支付 webhook→自动写入→展示 | 0 / 2 / 3 | — / 低 / 中 | Phase 1 零开发 |
| **参与资格校验** | 人工判断（谁可投按书面约定） | 可选：白名单校验 | 与论坛/签约系统关联，自动校验 | 0 / 1 / 3 | — / 低 / 中 | Phase 3 需用户身份打通 |

**困难点**：Phase 3 支付接入需合规资质（如 Stripe 需实体、支付宝需企业认证）；建国期若主体未就绪，可长期停留 Phase 1/2。

## 三、实现路径与方案

### 方案 A：最小可行（Phase 1，推荐建国期启动）

| 项目 | 实现 |
|------|------|
| **入账** | 参与者转账至约定账户；Founder 核对后，在论坛「建国者共建金」专帖追加一行：`参与人 | 日期 | 金额` |
| **出账** | 提议帖→CTO 确认回复→创世者决议（可 PDF 签署）→人工划款→在专帖追加出账记录 |
| **公示** | 专帖首楼或置顶维护表格：期初、入账合计、出账合计、期末；或使用 Notion/飞书表格，链接至论坛 |
| **CTO 工作量** | **0 人天**（无开发） |

### 方案 B：轻量系统化（Phase 2）

| 项目 | 实现 |
|------|------|
| **入账** | 同上；新增：简单表单（参与人、日期、金额）提交后写入 JSON 或 Supabase 表 |
| **出账** | 提议→确认→决议，各环节表单提交，状态写入 DB |
| **公示** | 静态页或简单 SPA，读 JSON/DB，展示入/出/余额表格；可挂 signa-website 子路径或独立页 |
| **CTO 工作量** | **3–6 人天** |

### 方案 C：完整集成（Phase 3）

| 项目 | 实现 |
|------|------|
| **入账** | 支付链接（Stripe/支付宝）→webhook→自动入账；或继续人工录入但表单化 |
| **出账** | 治理后台拨款流程；可选自动划款（需支付出款能力） |
| **公示** | GAIA 官网或治理后台「共建金」板块；与公益金池、建国债券等并列展示 |
| **CTO 工作量** | **8–15 人天**（含支付接入、治理工作流、官网集成） |

## 四、依赖与前置条件

| 依赖 | Phase 1 | Phase 2 | Phase 3 | 说明 |
|------|---------|---------|---------|------|

| **参与资格书面约定** | 是 | 是 | 是 | Founder；非 CTO |

| **支付资质（Stripe/支付宝等）** | 否 | 否 | 是 | Phase 3 支付接入需要 |
| **治理后台** | 否 | 否 | 可选 | Phase 3 若集成治理流程 |

## 五、工作量汇总

| 阶段 | CTO 开发人天 | 主要工作 | 可延后项 |
|------|--------------|----------|----------|
| **Phase 1** | 0 | 无；Founder 手工维护 | — |
| **Phase 2** | 3–6 | 入账/出账表单；公示页；JSON/轻量 DB | 导出、任务关联 |
| **Phase 3** | 8–15 | 支付接入；治理工作流；官网集成；可选自动划款 | 自动划款、参与资格自动校验 |

**建议**：建国期用 Phase 1 即可跑通机制、建立信任；待首笔采购完成、流程稳定后，再评估是否上 Phase 2。

## 六、风险与缓解

| 风险 | 缓解 |
|------|------|
| 人工录入错误 | Phase 1 双人核对；Phase 2 表单校验、必填项 |
| 公示滞后 | 约定更新频率（如每日/每笔后）；Phase 2 表单提交即触发更新 |
| 支付接入合规 | Phase 3 前确认主体、资质；或长期保持人工入账 |
| 与公益金池混淆 | 账户/科目隔离；展示时明确标注「建国者共建金」 |

---

## 七、与建国期人事网评估表的关系

| 机制 | 共享依赖 | 可复用 |
|------|----------|--------|
| 建国期人事网 | 论坛、任务清单 | 任务看板、任务清单 |
| 建国者共建金 | 论坛（Phase 1）、任务清单 | 动用提议时可关联任务；Phase 3 可复用治理后台 |

两项机制可并行推进；共建金 Phase 1 不阻塞人事网。
