# 建国期人事网：CTO 实现评估表

**用途**：从 CTO 视角评估建国期人事网各模块的实现路径、工作量与困难度；供排期与选型决策  
**依据**：《建国期人事网：自发协作结构设计（草案）》《创世贡献记录与确认机制》《GAIA 社区早期权重计算逻辑》

---

## 一、总览

| 阶段 | 实现路径 | 总工作量（人天） | 难度 | 对论坛的依赖 |
|------|----------|------------------|------|--------------|
| **Phase 1** | 论坛原生 + 手工看板 | 2–5 | 低 | 高（论坛需先就绪） |
| **Phase 2** | 论坛 API + 轻量脚本 + 展示页 | 10–20 | 中 | 高（需 API） |
| **Phase 3** | 数据层 + 计算引擎 + 图谱前端 | 30–50 | 高 | 中（可异步） |

## 四、依赖关系与推荐实施顺序

```
论坛就绪（Discourse / 自建）
    │
    ├─→ Phase 1：任务专区 + 认领帖 + 手工看板（Founder 主责，CTO 辅以论坛配置）
    │
    ├─→ Phase 2：论坛 API 对接 → 任务状态解析 → 看板自动化
    │             参与度聚合 → RP 计算脚本 → 展示页
    │             标签/徽章（若 Discourse）
    │
    └─→ Phase 3：数据层（贡献、任务、角色）→ RP 实时计算 → 衰减
                 图谱前端 → 裁定者池自动维护 → 按案遴选
```

**建议**：Phase 1 不依赖 CTO 开发，仅需论坛可用；Phase 2 在论坛 API 就绪后启动；Phase 3 可待建国 90 日结束、FCR 评估完成后再规划。

## 五、工作量汇总（按阶段）

| 阶段 | CTO 开发人天 | 主要工作 | 可延后项 |
|------|--------------|----------|----------|
| **Phase 1** | 0–2 | 论坛搭建/配置；可选：静态看板页 | 无 |
| **Phase 2** | 10–18 | 论坛 API 对接；任务解析；参与度/RP 脚本；展示页；标签配置 | RP 公式、有效邀请 |
| **Phase 3** | 25–40 | 数据层；RP 实时+衰减；图谱；裁定者池；按案遴选 | 按案遴选可人工先行 |

**总区间**：35–60 人天（含 Phase 3）；若只做 Phase 1+2，约 10–20 人天。

## 六、风险与缓解

| 风险 | 缓解 |
|------|------|
| 论坛无 API（如 GitHub Discussions） | Phase 2 改用人工维护 JSON + 静态页；或提前迁移 Discourse |
| RP 公式争议 | 先手工计算 1–2 轮，共识后再自动化 |
| 裁定者、门类理事会就位节奏不确定 | Phase 3 裁定者池、按案遴选可人工先行，系统后补 |
| 数据分散（论坛+表格+签约） | Phase 3 前确定统一数据入口；或接受多源手工汇总 |
