# Founder–沈炎君 协作与治理要点

**用途**：新对话中，AI 可据此调用、提醒、综合考量。  
**适用场景**：涉及 沈炎君、IT 协作、技术决策、贡献分配、FCR 时。

---

## 一、角色与决策权

| 角色 | 职能 | 原则 |
|------|------|------|
| **Founder** | 唯一决策者、拍板人 | 一个主体不能有两个头 |
| **AI** | 技术审查、合理性评估、向 Founder 汇报 | 不越权决策 |
| **沈炎君** | 执行者，可提出反馈 | 合理建议可采纳，最终决策在 Founder |

**流程**：沈炎君提交 → AI 审查（合规宪法/意志）→ Founder 拍板

---

## 二、数据同步与冲突

| 问题 | 方案 |
|------|------|
| 决策者与执行者数据不同步 | 单一数据源（Git / 云文档） |
| 冲突协调 | Founder 拥有规范类文档编辑权；沈炎君通过 PR/评论提议 |
| 共享范围 | 技术文档、Sprint 可共享；密钥、人际档案不共享 |

---

## 三、贡献与 FCR

| 结论 | 依据 |
|------|------|
| **原创思想来自 Founder** | 战略、产品、市场、投入 |
| **技术贡献可评估** | 含金量可量化，AI 时代实现层可替代性提高 |
| **不承诺「一半」** | 用 FCR 协议明确比例，不以口头对半分为准 |
| **「够不够」的锚点** | 收入达生活保障、家庭压力缓解、他感到被公平对待 |

**Founder 承诺**：应得的给你，欠你的也会补上；优先满足移民家庭需求。

---

## 四、心理预期与平衡

| 沈炎君可能期待 | 建议回应 |
|----------------|----------|
| 将来一半归他 | 用协议和收入体现价值，不口头承诺对半 |
| 技术贡献被低估 | 明确认可技术重要性，FCR 写进协议 |
| 将来被甩开 | FCR 长期有效，项目在则持续受益 |

**原则**：协议透明 + 收入优先 + 明确认可，比模糊承诺更利于友谊与合作。

---

## 五、网站协作分工（signa-website）

| 角色 | 负责 | 说明 |
|------|------|------|
| **沈炎君** | 设计、逻辑、后台接入 | 架构、数据结构、API、部署 |
| **Founder** | 前台呈现、文案内容 | 决策层：展示什么、说什么、语气如何 |
| **AI（Cursor）** | 前台文案与排版落地 | 执行层：直接修改 i18n、sections 等文件，按《网站文案与战略对齐指南》更新 |

**流程**：沈炎君搭建结构 → Founder 定方向与内容 → AI 在项目中改文件、及时同步  
**无需**：AI 与 Founder 反复沟通文案细节；AI 与沈炎君沟通逻辑/后台。  
**工具**：Git 单一数据源，共同完成。

## 六、账户与可见范围（如何同时协作且他看不到你全量数据）

### 6.1 账户

| 角色 | 需要 | 说明 |
|------|------|------|
| **沈炎君** | 独立账户 | 自己的 GitHub + Cursor（或 VS Code） |
| **你** | 现有账户 | 不变，GAIA_System_OS 在你名下 |

**结论**：他需注册自己的账户；不共享你的 Cursor/账号，否则他会看到人际档案、战略等全量数据。

### 6.2 可见范围

| 沈炎君可见 | 沈炎君不可见 |
|------------|--------------|
| signa-website 代码仓库 | GAIA_System_OS 全库 |
| 网站文案与战略对齐指南（放入 signa-website 内） | 人际关系、00_索引、宪法、FCR、战略方案全文 |
| 产品/技术相关必要文档 | 密钥、人物档案 |

### 6.3 实现方式：signa-website 独立为 Git 仓库

```
┌─────────────────────────────────┐     Git push/pull      ┌──────────────────────┐
│ 你（Cursor + GAIA_System_OS）   │ ◄──────────────────►  │  signa-website 仓库  │
│ - 全量数据在此                  │                        │  （GitHub/GitLab）    │
│ - signa-website 以 submodule    │                        └──────────┬───────────┘
│   或同一目录引用该仓库           │                                   │
└─────────────────────────────────┘                                   │
                                                                      │ Git clone
                                                                      ▼
                                                      ┌──────────────────────────┐
                                                      │ 沈炎君（独立 Cursor/IDE）│
                                                      │ - 仅 clone signa-website │
                                                      │ - 看不到你的其他数据     │
                                                      └──────────────────────────┘
```

### 6.4 前后台同步逻辑

| 你这边 | 沈炎君那边 |
|--------|------------|
| 在 GAIA_System_OS 中操作 signa-website（或 submodule） | 在他本地 clone 的 signa-website 中开发 |
| 与我讨论 → 我改 i18n、文案 | 他改逻辑、API、后台 |
| push 到 signa-website 远程仓库 | pull 你的改动，push 他的改动 |
| **共享界面**：同一 Git 仓库的 `common.json`、sections、数据结构 | 他定义 schema，你这边文案填入 |

**数据同步**：通过 Git 完成。他改 `products.tiers` 的 key → 你这边对应的 i18n 按新 key 调整；你改文案 → 他 pull 即得。

### 6.5 待执行步骤

1. **你**：将 signa-website 拆为独立 Git 仓库并推送到 GitHub/GitLab（若尚未如此）
2. **你**：把《网站文案与战略对齐指南》复制到 signa-website 内（如 `docs/`），供他参考
3. **沈炎君**：注册 GitHub、安装 Cursor 或 VS Code，clone 仅 signa-website 仓库
4. **协作**：双方 push/pull 同一仓库；冲突时按 §五 分工解决

**发给沈炎君的操作文档**：`04_工程与落地/signa-website/沈炎君-signa-website 协作操作指南.md`。他按此文操作即可；含益处说明、为何当前方案最佳、对其长期目标的尊重。

### 6.6 部署审核关口（实时掌握上线是否符合预期）

| 角色 | 职责 |
|------|------|
| **沈炎君** | 部署至 Staging → 提供 URL 给 Founder → 经审核通过后部署 Production |
| **Founder** | 将 Staging URL 交给 AI 审核 |
| **AI** | 按《部署审核关口》`04_工程与落地/signa-website/部署审核关口.md` 清单逐项检查，输出通过/不通过及问题清单 |

**触发方式**：Founder 说「审核 signa-website 部署，URL：xxx」或「部署审核，Staging：xxx」。

## 七、AI 调用提醒

当对话涉及以下任一关键词时，AI 须主动调用本要点：

- 沈炎君、CTO、技术合伙人
- IT 协作、数据同步、Git、共享
- FCR、贡献分配、技术含金量
- 决策权、拍板、冲突协调
- 友谊与利益、心理预期
- signa-website、前台文案、i18n、网站协作
- 账户、可见范围、Git 仓库拆分、沈炎君权限
- 部署审核、Staging URL、上线前审核
