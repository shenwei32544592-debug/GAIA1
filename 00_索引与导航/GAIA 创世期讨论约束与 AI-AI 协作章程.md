# GAIA 创世期讨论约束与 AI-AI 协作章程

**版本**：v1.0  
**日期**：2026-02-16  
**用途**：前置文档；约束讨论边界、防跑题、保证每轮产出可落地结论

---

## 一、当前阶段真实状态

GAIA 已完成叙事层、制度层、工程层、市场切口（AI 训练合规准入）的设计。尚未完成的关键跃迁：从「可读」到「可被采购/采纳」。

创世人所亟需的，不是再造宏大秩序，而是做出一个**「被对方内部流程接受」的最小产品化单元（MAU）**。

---

## 二、讨论有效性判断标准

每个讨论话题必须同时满足以下三项，否则判定为「未来主义跑题」：

| 条件 | 含义 |
|------|------|
| **A** | 直接服务于「首批采用」——能帮助拿下首家模型商/数据经纪商/行业协会的内部评审通过 |
| **B** | 可在 2–6 周内形成可交付物——输出必须能变成文档/API spec/demo repo/评审包/合同附件之一 |
| **C** | 可被第三方验证——不是「我们认为」，而是对方能验收 |

## 三、讨论范围的硬边界（必须暂停）

| 禁区 | 说明 |
|------|------|
| 全球司法互认/国际条约层面 | 除非为合规措辞最小化 |
| 完全链上化/DAO化/金融清算系统设计 | 与去金融化主线冲突 |
| 全网爬虫级规模化采集与实时索引 | 先做可控范围 PoC |
| 「法院一定会采信」式假设 | 只讨论「怎样提高被引用概率」 |
| 泛泛的「生态愿景」扩写 | 除非直接转化为销售/评审材料 |

---

## 四、当前阶段必争之点（MAU 组件）

任何工作必须落在以下四件事之一：

| 组件 | 说明 |
|------|------|
| **(1) Batch Compliance Report** | 输入数据集/URL/指纹；输出 clean/pending/locked 汇总；可审计字段；建议动作 |
| **(2) 内部合规 SOP 包** | Manual Addendum + Workflow Spec；系统接口点、故障处理、延迟容错 |
| **(3) 历史确定性与未来锁定的法务边界语言** | 不回溯何种范围；未来调整的触发条件；谁负责执行、谁免责 |
| **(4) Reference Implementation** | 最小可运行 demo；合规审计 + 日志生成；可被对方工程团队复制粘贴跑通 |

## 五、双 AI 协作方法论

### 5.1 角色分工

| 角色 | 职责 |
|------|------|
| **收敛器/交付物工程化** | 把讨论压成规范、模板、接口、条款、验收标准；负责「写成能用的文件」 |
| **扩张器/红队压力测试** | 专门找漏洞、误解点、法律风险、工程失败点、对方会如何反驳；负责「不丢人」 |

规则：扩张器只负责提出问题与风险；收敛器负责做决策与落款。

### 5.2 单回合讨论输出格式

每轮围绕一个点，双方必须输出：

- **结论**（1–2 句）：我们认定什么是对的
- **边界**（≤3 条）：我们明确不做什么
- **证据/理由**（最多 5 条）：为什么这样定
- **可交付物**（1 个）：这轮产出变成哪个文件/接口
- **验收标准 (DoD)**：如何判定完成（可测试）

### 5.3 收敛判定

当同时满足以下条件，宣布「暂时讨论好了」：

- 结论可写进合同/手册/API spec
- 红队提出的主要攻击点（≤3 个）已有明确处理
- 交付物的 Done Definition 可被第三方验收
- 下一步行动可在 2 周内推进（明确负责人/文件名/接口名）
