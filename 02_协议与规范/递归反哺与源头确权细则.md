## 一、正当性前提：自发组织与自我治理

**本协议规定的版权保护，其正当性来自：**

艺术家带头承认共识，主动执行并自我管理他们共同参与制定的协议——这是自发组织与自我治理的产物，而非外部强加。

| 要素 | 含义 |
|------|------|
| **自发组织** | 参与者自愿加入，非强制纳入 |
| **除现世版权法外** | 在现行法律之外，增加协议内共同约定 |
| **艺术家带头承认** | 艺术家是首先认可并践行共识的主体 |
| **自己成立的协议** | 协议由参与者共同制定，非平台或政府单方颁布 |
| **执行与自我管理** | 参与者按共识执行，自我治理 |

**规矩的层级**：宪法是根本大法；递归反哺、FCR 等是宪法下的具体细则。加入联盟，即表示认同宪法和这些细则，并愿意照做。

**参与即承诺**：加入我们，即表示认同上述共识并愿意照做。

## 二、源头两层结构

任何版权在协议内最多同时拥有**两层**溯源关系，不得向上越级。

### 2.1 绝对源头

| 维度 | 说明 |
|------|------|
| **定义** | 真正的原创者；可能当下未知 |
| **可修正性** | 可经未来**审查动议**（公众指认等）确认或更正 |
| **当前权利人** | 可能知情或不知情自己非真正源头，均可接受审查动议 |
| **层级** | 最上游；认定即止，不再追溯 |

### 2.2 应用层源头

| 维度 | 说明 |
|------|------|
| **定义** | 在绝对源头影响下创作、**首次在平台登记**的版权主体 |
| **可确权** | 平台可验证、可追溯 |
| **层级** | 平台运作中的「可见源头」 |
| **备选称法** | 登记源头 / 平台源头 / 确权源头 |

### 2.3 两层约束

- 任何版权**最多同时**拥有：绝对源头 + 应用层源头
- **不再向上越级**：认定到绝对源头即止

## 三、收益回流（递归反哺）实现规则

### 3.1 回流来源

| 原则 | 说明 |
|------|------|
| **来源** | 必须从**经认定**的、在平台销售的、使用源头创意的作品的版权收益中拆分 |
| **扣减对象** | 从衍生创作者的 **65% 艺术家利润部分** 中抽取；抽取比例为**暂定**，见下文 §4.3 |
| **责任归属** | 使用源头创意的艺术家须让渡部分收益给源头。这是必须履行的责任，也是对每一位源头——包括可能成为源头的自己——的致敬，对艺术创新的鼓励与肯定；敬重源头，即敬重自己、敬重艺术家这个身份 |

### 3.2 生效时点

- **自认定日起执行**：认定日及之后的销售，按回流规则执行。
- **认定日前已完成的销售不追溯**：认定日之前的销售，不补扣、不追回。

**举例**：

| 情形 | 说明 | 举例 |
|------|------|------|
| 认定日 | 某日经共识或程序确认：B 的作品使用了 A 的创意 | 2025 年 3 月 15 日认定 |
| 须回流 | 认定日**及之后**的销售，按规则向 A 回流 | 3 月 16 日、3 月 20 日卖出的订单 → 回流给 A |
| 不追溯 | 认定日**之前**已完成的销售，不补扣、不追回 | 3 月 1 日、3 月 10 日卖出的订单 → 不回流，钱归 B |

### 3.3 资金流示意

```
衍生订单：B 的作品经认定使用了 A 的创意
       │
       ├─ 65% 艺术家利润部分（B 的份额）
       │       ├─ 按 A 的定价 → 回流给 A（绝对源头或应用层源头）
       │       └─ 剩余 → B
       │
       ├─ 创世分润池（2%+2%+1%；Anchor）
       ├─ 20% 公共运营池（含回流机制建设与运维）
       └─ 10% 风险储备金
```

### 3.4 建设费用

- **回流给艺术家的钱**：来自衍生销售本身，不从其他池抽调。
- **建设回流机制的钱**：来自公共运营池 (20%)。CTO 负责落实任何节点提出的 IT 建设需求，保障联盟与时俱进、高效先进；职责详见《CTO 技术共识协议》《核心技术节点合作备忘录》等。

## 四、版权使用费定价权

### 4.1 定价权归属

| 原则 | 说明 |
|------|------|
| **归属** | 由**版权所有者**自主定价（本体系中指绝对源头或应用层源头） |
| **策略** | 想让更多人用 → 定低（如采用协议默认 0.1%）；不想让人用 → 定高（如 50% 以上有效阻止借用） |
| **平台角色** | 提供定价入口与执行，不代为定价；源头不设定时采用协议默认，见 §4.3 |

### 4.2 定价形式

| 形式 | 说明 | 举例 |
|------|------|------|
| **比例** | 按每次衍生销售的百分比 | 如 0.1%、1%、5%；未设定时见 §4.3 默认比例 |
| **固定金额** | 每次销售固定金额 | 如每次 $2、$10 |
| **组合** | 比例 + 保底/封顶 | 如 1% 或至少 $5，取高者 |
| **0%** | 无偿开放；须治理允许 | 免费让人用，表示对传播的鼓励 |

### 4.3 协议默认比例

当绝对源头**未设定**使用费时，采用协议约定默认比例：

| 项目 | 约定 |
|------|------|
| **默认值** | 衍生创作者 艺术家利润部分（65%）的 **0.1%**（**暂定比例**） |
| **性质** | 象征性最低回流，保证「必有敬意」 |
| **后续** | 以后根据众人讨论后共识，再明确确定最终比例 |
| **可调** | 比例可经共识程序修订 |

**计算示例**：Net Pool $1000，65% = $650；暂定 0.1% × $650 ≈ $0.65。

### 4.4 多源引用

当衍生作品同时引用多个源头时：

| 规则 | 说明 |
|------|------|
| **各自定价** | 各源头按其事先设定的价格适用 |
| **总回流封顶** | 回流总额 ≤ 衍生创作者 65% 的 50% |
| **超限处理** | 按各源头报价比例等比缩减，使总和 = 封顶 |
| **争议** | 对引用范围、比例有异议时，由**裁定者**裁定 |

**IT 与艺术家实现**：算法可自动化——订单关联多源头及各自价格 → 分配引擎加总 → 若超封顶则等比缩减 → 生成 RuleID。艺术家须在发布衍生作品时声明引用了哪些源头；平台提供勾选/关联入口。争议由裁定者裁定，属共识流程，非系统自动。无根本实现障碍；多源引用若初期少见，可先支持单源，多源作为后续迭代。

## 五、行业始祖 0.1%：待认领池与处分

### 5.1 适用场景

当某**品类**存在共识上的「行业始祖」或「老祖宗」，但**无法指认**在世具体权利人时，衍生创作者可声明「归属某品类传统，无具体可指认源头」，触发本机制。

| 情境 | 说明 |
|------|------|
| **传统行业** | 如书法、非遗、某工艺门类；品类由门类理事会维护 |
| **共识** | 行业皆知有源头，但无人可确认 |
| **原则** | 0.1% 先暂存，支持迟到正义；钱不进公益直至业内认定无法指认 |

### 5.2 待认领池：暂存优先，不入公益

| 项目 | 约定 |
|------|------|
| **比例** | 与协议默认比例一致（65% 的 0.1%，暂定） |
| **流向** | **行业始祖待认领池**（非公益金池）；按品类分账 |
| **声明** | 衍生创作者声明「归属某品类传统，无具体可指认源头」时触发 |
| **性质** | 暂存；不直接进入公益金池；预留給将来可能被指认的始祖 |

**品类**：粗粒度由门类理事会定义（如书法、剪纸、刺绣等）；声明时须选定品类；无法精确对应时可选「综合传统」。

### 5.3 指认与支付：找到始祖

经**公众指认（PAC）**或等效程序指认某人为某品类始祖并确认后：

| 环节 | 约定 |
|------|------|
| **绑定账户** | 该人绑定收款账户 |
| **支付范围** | 自确认之日起：**该品类历史积累** + **未来全部 0.1%**（永久） |
| **资格终止** | 直至行业内公投取消其资格（见 §5.4） |
| **钱不进公益** | 此路径下 0.1% 支付给始祖，不进入公益金池 |

### 5.4 公投取消资格

行业内公投可**取消**某始祖的接收资格（如发现指认错误、造假等）。

| 后果 | 约定 |
|------|------|
| **已发放** | 原则上不溯及追回（执行成本高）；欺诈等另按单独程序 |
| **未来 0.1%** | 停止向该人支付；回流该品类待认领池 |
| **后续** | 可再次指认新始祖；或经公投转入公益（见 §5.5） |

### 5.5 转入公益

仅在以下情形下，某品类待认领余额**转入**公益金池的行业始祖部分：

| 情形 | 约定 |
|------|------|
| **公投认定** | 行业内公投认定「该品类无法指认具体始祖」 |
| **长期无指认** | 某品类待认领存在满 N 年（具体年数由治理约定）且无有效指认，可经公投转入 |
| **用途** | 转入后按协议约定定向用途（传承、教育、扶持新人、记录）；平台级治理决议动用 |

**待认领池处分三种去向**：① 找到始祖 → 支付该人（历史积累 + 未来 0.1%）；② 公投认定无法指认或长期无指认 → 转入公益金池；③ 未决则暂存，等待将来指认。处分权在行业内公投与公众指认，非平台独家。

### 5.6 与可指认源头的关系

- 若衍生作品**仅归属某品类、无具体可指认源头** → 0.1% 进入**待认领池**（该品类）
- 若**可指认**具体源头 → 按该人定价或默认支付该人，**不**同时计入待认领池
- 若公众指认事后指认出具体源头 → 改向该人支付；若该品类在待认领池有积累，可一并支付

### 5.7 目的

| 目的 | 说明 |
|------|------|
| **迟到正义** | 为将来可能被指认的始祖保留份额，不提前划入公益 |
| **心存感激** | 承认不可指认源头的存在，不把收益尽归己有 |
| **行业自治** | 指认、取消资格、转入公益，均由业内共识或公投决定 |

**IT 与制度实现可行性**：0.1% 分流、按品类计入待认领池、指认后支付始祖、公投转入公益——均为**标准账务逻辑 + 治理决议驱动**。订单分配时按衍生创作者声明的品类，将 0.1% 计入该品类子账户；经公众指认确认始祖后，系统将待认领余额及未来 0.1% 路由至其账户；公投转入公益时，治理决议触发系统划款。与电商平台的退款池、代管金、多级分账无异；表结构见 §5.8.6。**无根本实现障碍**；想法可落地。详见《GAIA 技术可行性说明（公众版）》§7.3。

### 5.8 治理与财务管理

#### 5.8.1 治理结构

| 环节 | 主体 | 职责 |
|------|------|------|
| **决策** | 平台级治理（Domain Council 或行业代表联合） | 决定基金用途、拨款方案、年度计划 |
| **托管** | 平台（LLC）或指定合规主体 | 持有资金、执行拨款、维护账簿 |
| **执行** | 平台财务按决议划款 | 无平台级批准决议不划款 |
| **审计** | 第三方或治理推举的审计人 | 年度审计；可随时查阅流水与余额 |

**决策与执行分离**：治理主体不直接持有现金，仅通过批准用途与拨款方案行使治理权；平台作为托管方按决议执行。

#### 5.8.2 资金存放：公益金池与待认领池

**原则**：平台内有明确记录；动用权由协议预先定好；不可能超支；余额永不为负；全平台可共享查看明细账。

| 层级 | 做法 |
|------|------|
| **行业始祖待认领池** | 独立逻辑分区或子账户；0.1% 按品类暂存；**不入公益**直至公投转入；可支付给被指认始祖 |
| **公益金池（全平台公益积累金）** | 全平台**仅一个**公益金银行账户或公益信托；详见[《公益金池细则》](公益金池细则.html) |
| **入池来源** | ① 行业始祖部分（从待认领池**转入**，经公投认定无法指认或满 N 年无指认后）；② 1% 名誉成本（创世者与创世技术官各认捐 0.5%；创世者提议权与决策权专属；创世技术官知情权与建议权、无决定权；其它节点提议权与监督权）；③ 主动捐赠（任何人可捐，含客户与联盟任意角色；平台须提供捐赠通道；捐赠可获公益勋章，勋章持有人享有代表联盟出席公益活动时的优先参与权；详见《公益金池细则》§2.1） |
| **收益归集** | 上述来源统一入池，计入基金可用余额；入账记录标注来源类型 |
| **平台账簿** | 入账、出账、可用余额均有明确记录；入账可追溯至 OrderID/来源，出账可追溯至 ResolutionID |
| **动用权** | 见 §5.8.2.1 动用权分层 |
| **不可超支** | 单笔拨款 ≤ 可用余额；系统校验；余额永不为负 |
| **透明** | 全平台参与者可共享查看明细账；入账、出账、用途、余额皆可查 |

**特性**：账户唯一、记录明确、动用权已定、不可能超支、余额永不为负；明细账全平台共享可查。

##### 5.8.2.1 动用权分层

公益金池内资金按**决策权**分为两类：

| 类型 | 来源 | 决策主体 | 说明 |
|------|------|----------|------|
| **协议约定定向用途** | 行业始祖部分（从待认领池转入后的 0.1%） | 平台级治理（Domain Council 或行业代表联合） | 用途为传承、教育、扶持新人、记录等；须经平台级拨款决议 |
| **创世者决策权部分** | 1% 名誉成本 | **创世者（Founder）专属** | 用途须符合 §5.8.2.2 边界；受披露与审计约束（§5.8.2.3）；制度性影响力，非个人意志（§5.8.2.4） |

**要点**：同一池、同一银行账户；平台账簿按**决策权类型**分设虚拟子余额（行业始祖部分 / 创世者决策权部分），动用时校验：平台级决议仅可从行业始祖部分划出；**创世者（Founder）**独家决议仅可从创世者决策权部分划出。**CTO 知情权**详见 §5.8.2.1a。CTO 无决定权。二者互不挪用。**待认领池**独立记账，不参与公益金池动用权分层。

##### 5.8.2.1a CTO 知情权：程序与「决议须注明是否采纳」

创世者动用 1% 创世者决策权部分前，须履行以下程序；**拨款决议**中须对 CTO 建议的处理作出明确记录。

| 环节 | 要求 |
|------|------|
| **提前披露** | 创世者须提前 **7 天** 向 CTO 披露用途草案（用途、金额、收款方等） |
| **CTO 建议** | 7 天内 CTO 可提交**书面建议**（如：用途是否妥当、金额建议、风险提示等）；可不提 |
| **决议注明** | 创世者签发拨款决议时，须在决议正文中注明：① CTO 未提出建议，或 ② 已听取 CTO 建议，并写明「已采纳」或「未采纳，理由：[简要说明]」 |

**「决议须注明是否采纳」含义**：拨款决议（含 ResolutionID、用途、金额、收款方）为可追溯的法律与审计依据。决议须载明对 CTO 建议的处理——若 CTO 有建议而未采纳，须写明理由，形成可查记录；不改变创世者最终决策权，但增强透明与可审查性。

**举例**：

| 情境 | CTO 建议 | 决议中须注明 |
|------|----------|--------------|
| CTO 建议先核实收款方资质 | 「建议先核实该组织资质再拨款」 | 采纳 → 「已采纳 CTO 建议；已核实资质后拨款」；不采纳 → 「未采纳 CTO 建议，理由：该组织为平台既有合作方，资质已备查」 |
| CTO 建议降低金额 | 「建议将金额从 $5000 降至 $3000」 | 采纳 → 「已采纳 CTO 建议，按 $3000 拨款」；不采纳 → 「未采纳 CTO 建议，理由：该笔为年度协议拨款，金额已在协议中约定」 |
| CTO 未提建议 | （7 天内未提交） | 「CTO 未提出建议」 |

##### 5.8.2.2 创世者（Founder）决策权：用途边界

| 条款 | 内容 |
|------|------|
| **用途范围** | 创世者决策权部分的拨款，用途须限于：公益、教育、行业扶持、新人孵化、记录与传承、联盟内成员人道救助（10% 风险储备金结余不足或暂不可用时的补充）、符合平台使命的其它公益性质用途 |
| **联盟内成员界定** | 曾在体系内拥有账户或完成宪法承认行为的自然人/实体；认定以系统可追溯记录为准。详见《GAIA 标准术语与概念表》 |
| **成员救助专项约束** | 仅限重大人道危机（重大疾病、自然灾害、意外致重大损失）；须先向 10% 池申请或确认 10% 不可用；年度占比 ≤ 该部分当年入账 20%；须可核验证明；标准公开。详见《公益金池细则》§4.3.1 |
| **禁止用途** | 不得用于创世者个人消费、与平台无关的商业投资、政治捐献、或任何可能损害平台声誉的用途 |
| **单笔上限** | 单笔拨款 ≤ 该部分当期余额的 30%，或 ≤ 过去 12 个月该部分入账总额的 50%，取二者较低值；经裁定者或门类理事会 事前书面豁免可例外 |
| **年度复核** | 每财年结束后 30 日内，创世者须向治理披露该部分当年拨款汇总（用途、金额、收款方）；可纳入季度披露一并公布 |
| **大额披露** | 单笔动用 **≥ 该部分当期余额 20%** 时，须在季度披露中**单独列出**用途与决策依据；全平台可查 |

##### 5.8.2.3 创世者决策权：制衡与审计

| 条款 | 内容 |
|------|------|
| **定期披露** | 创世者决策权部分的入账、出账、余额，与行业始祖部分一并列入季度披露；全平台可查 |
| **审计权** | 年度审计须覆盖创世者决策权部分；审计方有权查阅该部分完整流水、决议记录及用途说明 |
| **抽查权** | 裁定者或门类理事会 可随时提议对该部分拨款进行抽样复核；创世者须在合理期限内提供用途合理性说明 |
| **救济** | 若经共识程序认定某笔拨款违反用途边界或禁止用途，可要求创世者等额补回该部分；补回金额从创世者后续应得收益中扣减 |

##### 5.8.2.4 创世者决策权：制度性澄清

1% 公益金用途的**提议权与最终决策权**为创世者（Founder）**专属**；可主动提议亦可最终决定；此为创世者制度性影响力来源。创世技术官有知情权与建议权，无决定权；其它节点有提议权与监督权。该权力为**协议约定的制度性权力**，非个人意志或软性影响力。其行使须：
- 形成书面拨款决议（含用途、金额、收款方、决策日期）
- 关联 ResolutionID，可追溯、可审计
- 接受上述用途边界、披露与审计约束

该权力来源于创世者对 1% 名誉成本的认捐与贡献；二人各出 0.5%，但**用途决策权仅归属创世者**，系其核心制度优势。权力行使须与协议一致，不得超越约定边界。决策权**可移交**：死亡、失能时按承继规则；创世者自愿时可将权力交出予联盟内委员会。移交方式与原因、好处见《公益金池细则》§4.4。

#### 5.8.3 入账规则（公益金池）

| 项目 | 规则 |
|------|------|
| **触发** | 衍生创作者声明「归属某品类传统，无具体可指认源头」的订单完成结算后 |
| **金额** | 该订单 65% 艺术家利润部分 的 0.1%（暂定比例，与协议默认一致） |
| **流向** | ① 0.1% 触发时 → **待认领池**（按品类）；② 经公投转入后 → 公益金池行业始祖部分 |
| **记录** | 待认领池入账含：OrderID、品类、金额、结算日期；公益金池入账含来源（待认领转入/1%/捐赠） |

#### 5.8.4 出账规则

**待认领池出账**（见 §5.3、§5.5）：① 指认始祖并确认 → 支付该品类历史积累及未来 0.1% 至其绑定账户；② 公投转入公益 → 该品类余额划入公益金池行业始祖部分。

**公益金池出账**：

| 项目 | 规则 |
|------|------|
| **前置条件** | 须有对应决策主体的**拨款决议**（含用途、金额、收款方） |
| **决策主体校验** | 平台级治理决议 → 用途须为传承/教育/扶持新人/记录，金额 ≤ 行业始祖部分余额；创世者决议 → 金额 ≤ 创世者决策权部分余额 |
| **超支校验** | 决议金额 ≤ 对应部分可用余额；否则系统拒绝执行 |
| **执行** | 平台财务按决议划款；划款记录关联 ResolutionID 与决策主体 |
| **禁止** | 无决议不划款；越权决议不执行；超额不划款 |
| **披露** | 每笔出账及用途在治理披露中公开 |

#### 5.8.5 披露与审计

| 项目 | 要求 |
|------|------|
| **定期披露** | 每季度公布：公益金池期初/入账/出账/期末；待认领池各品类余额 |
| **审计** | 年度审计；审计方有权查阅完整流水与决议记录 |
| **留存** | 流水与决议至少保留 7 年 |

#### 5.8.6 IT 实现方向（供工程参考）

| 组件 | 要点 |
|------|------|
| **资金层** | 单一银行账户或公益信托；实际划款均从此出 |
| **待认领池** | `ancestor_escrow(category_id, amount, order_id, timestamp)`；按品类分账 |
| **账务表** | `fund_ledger(source_type, entry_type, amount, order_id/resolution_id, timestamp)`；`source_type` = 行业始祖转入 / 1%名誉成本 / 主动捐赠 |
| **虚拟子余额** | `balance_industry_fund`（从待认领转入部分）, `balance_founder_discretion` 由流水聚合 |
| **多源拆分引擎** | 0.1% 触发 → 待认领池；转入后 → 行业始祖部分；1%与捐赠 → 各路由 |
| **拨款校验** | 平台级决议：`amount ≤ balance_industry_fund` 且 `purpose IN (传承,教育,扶持新人,记录)`；创世者决议：`amount ≤ balance_founder_discretion` |
| **治理决议表** | `fund_resolution(resolution_id, decision_maker, purpose, amount, payee, status, cto_advice_status, cto_advice_note)`；创世者决议须含 `cto_advice_status`（未提建议/已采纳/未采纳及理由） |
